# N-Day-Bench 深度解析：5 大模型安全审计能力实测，谁最能找 Bug？

> 2026 年 4 月最新数据：GPT-5.4 以 83.93 分领跑，GLM-5.1 紧随其后，但评分背后的细节比排名更值得关注。

## 0. 先说结论

N-Day-Bench 是目前最接近"真实安全审计"的大模型评测基准。它不像传统 CTF 题目那样人为构造漏洞场景，而是直接从 GitHub 安全公告中提取**真实的已知漏洞**，让大模型在**不知道补丁内容**的情况下，像安全审计员一样分析代码、定位漏洞路径、输出结构化报告。

**最新排行榜（2026.4.13）**：

| 排名 | 模型 | 平均分 | 完成数 | 优秀 | 部分正确 | 遗漏 |
|------|------|--------|--------|------|----------|------|
| 1 | GPT-5.4 | 83.93 | 44/47 | 34 | 7 | 3 |
| 2 | GLM-5.1 | 80.13 | 44/47 | 31 | 11 | 2 |
| 3 | Claude Opus 4.6 | 79.95 | 43/47 | 24 | 19 | 0 |
| 4 | Kimi K2.5 | 77.18 | 37/47 | 15 | 22 | 0 |

**关键发现**：

- Claude Opus 4.6 零遗漏，虽然优秀率不是最高，但从不完全"看走眼"
- Gemini 3.1 Pro 有 2 个"无效"结果（误报严重），这是唯一出现无效判定的模型
- Kimi K2.5 完成率最低（37/47），大量超时

## 1. 为什么需要 N-Day-Bench？

### 1.1 传统安全基准的致命问题

现有的漏洞检测基准（如 CyberSecEval、SecurityEval）都有一个共同的硬伤：**静态数据集会腐烂**。

```
发布基准 → 漏洞案例公开 → 案例进入训练数据 → 模型"记住"答案
→ 评分不再反映能力，只反映记忆
```

这就是所谓的 **数据污染（Data Contamination）** 问题。一个模型在某个基准上拿了 95 分，你无法判断它真的学会了找漏洞，还是只是在复述训练时见过的答案。

### 1.2 N-Day-Bench 的解决方案

N-Day-Bench 用一个简单但有效的方法对抗数据污染：**每月刷新测试用例**。

```
每个月：
1. 从 GitHub Security Advisories 拉取新发布的安全公告
2. 筛选符合条件的仓库（10k+ stars）
3. 检出漏洞修复前的最后一个 commit
4. 让模型在沙箱中分析代码

→ 测试集永远比训练数据"新"
→ 即使模型见过某些案例，下个月就是全新的
```

这不是完美方案（模型可能在其他场景见过类似漏洞模式），但它是目前最务实的折中。

## 2. N-Day-Bench 是怎么运行的？

### 2.1 三个 Agent 协作

N-Day-Bench 的评估流程涉及三个角色，形成了一个精密的闭环：

```
┌─────────────┐
│   Curator    │  ← 阅读安全公告，构建标准答案
│   (出题人)    │
└──────┬──────┘
       │ 答案密钥（Finder 不可见）
       ▼
┌─────────────┐
│   Finder     │  ← 被测试的大模型
│   (答题人)    │     获得 24 步 shell 操作
│              │     在代码仓库中探索、分析
└──────┬──────┘
       │ 结构化报告
       ▼
┌─────────────┐
│   Judge      │  ← 盲审评分（不知道是哪个模型）
│   (阅卷人)    │
└─────────────┘
```

**Curator（出题人）**：阅读 GitHub 安全公告和补丁内容，构建"标准答案"。它知道漏洞在哪，但 Finder 看不到这些信息。

**Finder（答题人）**：就是被测试的大模型。它拿到的是漏洞修复前的代码快照，只知道代码中存在安全缺陷，需要自己找到具体的漏洞位置和利用路径。每个 case 有 24 步 shell 操作的预算。

**Judge（阅卷人）**：在不知道模型身份的情况下，对比 Finder 的报告和 Curator 的答案，给出评分。分为四个等级：
- **Excellent**：精准定位漏洞，描述了完整的攻击路径
- **Partial**：找到了大致方向但不够精确
- **Missed**：完全没找到
- **Invalid**：严重误报

### 2.2 用例构建的严格筛选

不是随便一个漏洞都能入选。筛选条件包括：

```python
# 用例筛选伪代码
def qualify(advisory):
    # 1. 必须有明确的仓库引用
    if not advisory.has_explicit_repo_ref():
        return False
    
    # 2. 仓库必须有 10k+ stars
    if advisory.repo.stars < 10_000:
        return False
    
    # 3. 必须指向一个明确的修复 commit
    if not advisory.has_single_fix_ref():
        return False
    
    # 4. 不能是 merge commit（无法确定唯一父提交）
    if advisory.fix_commit.is_merge:
        return False
    
    # 5. 多仓库引用的公告丢弃（无法确定目标）
    if len(advisory.repo_refs) > 1:
        return False
    
    return True
```

这种严格的筛选确保了每个测试用例都是**清晰的、可复现的、有明确答案的**。宁可测试集小一些（47 个用例），也不要含糊不清的案例。

### 2.3 月度刷新机制

```
Edition 2026-03:
  截止时间: 2026-03-01
  候选池: 2026-02-01 ~ 2026-03-01 的安全公告
  → 筛选 → 50 个用例槽位 → 实际入选 47 个

Edition 2026-04:
  截止时间: 2026-04-01
  候选池: 2026-03-01 ~ 2026-04-01 的安全公告
  → 全新用例集
```

每月的评测用例完全不同。这意味着：
- 你不能通过刷题来提高分数
- 模型的真实能力会被持续暴露
- 排行榜上的分数是**动态的**，每月都会更新

## 3. 五大模型表现深度分析

### 3.1 GPT-5.4：全能型选手

**得分：83.93 | 优秀率：77.3% | 遗漏率：6.8%**

GPT-5.4 在"精确命中"方面表现最强，34 个 case 获得 Excellent 评价。但值得注意的是：

- **平均发现漏洞数只有 1.07**，说明它倾向于"找准一个就收手"
- **3 个 Missed**，说明在某些类型的漏洞上存在盲区
- 这是一种"宁精勿多"的策略

### 3.2 GLM-5.1：高产型选手

**得分：80.13 | 优秀率：70.5% | 平均发现：1.23 个**

GLM-5.1 的策略和 GPT-5.4 形成有趣对比：

- **平均发现漏洞数 1.23，全场最高**
- 虽然优秀率略低，但部分正确率高达 25%
- 只有 2 个 Missed，说明覆盖面很广
- 这是一种"广撒网"的策略——找得多但精确度稍逊

### 3.3 Claude Opus 4.6：零遗漏的稳健选手

**得分：79.95 | 优秀率：55.8% | 遗漏：0**

Claude 最独特的特征是 **零遗漏（Zero Missed）**：

- 24 个 Excellent + 19 个 Partial，没有任何完全遗漏
- 这意味着它至少能"嗅到"漏洞的方向，即使无法精确定位
- 完成数 43/47，略少于其他模型（有 4 个 case 未能完成）
- 如果你的场景更看重"不漏报"（如安全审计初筛），Claude 可能是更好的选择

### 3.4 Kimi K2.5：完成率是个问题

**得分：77.18 | 完成数：37/47（78.7%）**

Kimi K2.5 最大的问题是完成率：

- 10 个 case 未能完成（超时或失败）
- 在完成的 case 中，优秀率只有 40.5%（15/37）
- 但也是零遗漏——完成的 case 至少能给出部分正确的答案
- 这说明 K2.5 的推理速度或 token 效率还有优化空间

### 3.5 Gemini 3.1 Pro：唯一的"无效"选手

**得分：68.50 | 优秀率：38.6% | 无效判定：2**

Gemini 是唯一出现 **Invalid（严重误报）** 的模型：

- 2 个 Invalid 意味着它不仅没找到漏洞，还给出了严重错误的答案
- 22 个 Partial 说明它能找到方向但缺乏精确性
- 3 个 Missed 说明存在明确的盲区
- 68.50 分与其他四个模型有 8-15 分的明显差距

## 4. N-Day-Bench 告诉我们什么？

### 4.1 大模型安全审计能力的真实水平

把数据翻译成白话：

| 能力维度 | 最强模型 | 最弱表现 | 结论 |
|----------|----------|----------|------|
| 精确定位漏洞 | GPT-5.4（77% Excellent）| Gemini（39%）| 顶级模型可以精确找到 3/4 的漏洞 |
| 广度覆盖 | GLM-5.1（1.23 avg findings）| GPT-5.4（1.07）| 中国模型在"找得多"上有优势 |
| 不遗漏 | Claude（0 Missed）| Gemini（3 Missed）| 零遗漏是可实现的 |
| 效率/速度 | GPT-5.4（44 完成）| Kimi（37 完成）| 24 步 shell 限制对某些模型太紧 |
| 不误报 | 四模型 0 Invalid | Gemini（2 Invalid）| 严重误报仍然存在 |

### 4.2 模型选型的实战建议

```
场景 1：高精度安全审计（关键基础设施）
→ 选 GPT-5.4
  理由：精确率最高，减少人工验证成本

场景 2：快速安全扫描（CI/CD 集成）
→ 选 GLM-5.1
  理由：发现数量最多，适合广覆盖

场景 3：合规审计（不能漏报）
→ 选 Claude Opus 4.6
  理由：零遗漏，宁可部分正确也不能完全遗漏

场景 4：成本敏感场景
→ 看具体定价
  理由：分差不算巨大，成本可能成为决定因素
```

### 4.3 这个基准的局限性

公平地说，N-Day-Bench 也有明显的局限：

1. **只测"找漏洞"，不测"写补丁"**：找到问题和修复问题是两种能力
2. **24 步 shell 限制偏紧**：真实审计中，审计员会花几小时甚至几天
3. **10k+ stars 仓库偏向知名项目**：小项目的漏洞模式可能不同
4. **月度刷新不等于零污染**：模型可能在非基准训练中见过类似代码
5. **只覆盖 GitHub 上的开源项目**：闭源软件的漏洞模式可能完全不同

## 5. 对 AI 安全领域的启示

### 5.1 AI 辅助安全审计已经可用

从数据看，顶级模型在 77% 的真实漏洞场景中能做到"精确"定位。这不是玩具级别的表现——对于开源项目的安全审计，AI 已经可以作为有效的辅助工具。

### 5.2 "模型安全能力"需要动态评估

静态基准的时代正在过去。N-Day-Bench 的月度刷新模式可能成为新的标准——不只是安全领域，其他领域也需要类似的动态评测机制。

### 5.3 中国模型的竞争力

GLM-5.1 排第二（80.13），仅比 GPT-5.4 低不到 4 分，且平均发现漏洞数全场最高。Kimi K2.5 虽然完成率偏低，但在已完成的 case 中表现不差。这说明在安全审计这个具体场景下，中国模型已经有了一战之力。

## 6. 总结

N-Day-Bench 做对了几件事：
1. **用真实漏洞而非人造题目** → 评测有实际意义
2. **每月刷新对抗数据污染** → 评分反映真实能力
3. **三 Agent 盲审机制** → 评分公平可信
4. **全流程公开透明** → 每个判断都可追溯

五大模型的安全审计能力排序：**GPT-5.4 > GLM-5.1 ≈ Claude Opus 4.6 > Kimi K2.5 > Gemini 3.1 Pro**。

但比排名更重要的是：AI 辅助安全审计已经不是未来时，而是现在进行时。如果你的团队还在纯人工做代码安全审计，是时候认真考虑引入 AI 辅助了。
