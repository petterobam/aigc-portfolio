## 实现方案：三步搞定自动周报

### Step 1: 数据源接入（1 分钟配置）

OpenClaw 支持从多个数据源自动提取工作内容：

**Git 提交记录**
```bash
# OpenClaw 会自动获取本周的 Git 提交记录
# 包含：提交时间、提交信息、修改文件、代码行数
```

**Jira 任务管理**
```bash
# OpenClaw 会自动同步 Jira 任务状态
# 包含：任务名称、任务状态、完成时间、负责人
```

**邮件/会议记录**
```bash
# OpenClaw 会从邮件和会议记录中提取关键信息
# 包含：会议决议、待办事项、协作情况
```

**配置示例**：
```yaml
# config/weekly-report.yaml
data_sources:
  git:
    enabled: true
    repos:
      - path: ~/projects/backend
      - path: ~/projects/frontend
    time_range: "week"

  jira:
    enabled: true
    url: "https://your-company.atlassian.net"
    project: "PROJ"
    time_range: "week"

  email:
    enabled: true
    filters:
      - from: "project-manager@company.com"
      - subject: "周会纪要"
    time_range: "week"
```

## 核心价值：为什么 OpenClaw 生成的周报更好？

### 1. 数据驱动，不是"拍脑袋"

**手动周报**：
"本周完成了用户认证模块重构，性能有所提升。"

**自动周报**：
"本周完成了 [PROJ-123] 用户认证模块重构，代码提交 12 个，新增代码 1,234 行。登录 API 响应时间从 2.3s 降低到 0.8s（降低 65%），错误率从 0.3% 降低到 0.1%。"

**区别**：
- ✅ 有具体数字（提交数、代码量、性能指标）
- ✅ 有任务关联（Jira 编号）
- ✅ 有对比数据（本周 vs 上周）

### 2. 条理清晰，不是"流水账"

**手动周报**：
```
本周做了很多事情，修复了一些 bug，优化了接口，还开了几次会。
下周计划完成订单重构和支付集成。
```

**自动周报**：
```
## 一、本周工作进展
### 1.1 后端开发
### 1.2 Bug 修复

## 二、数据统计
### 2.1 代码提交
### 2.2 任务完成
### 2.3 Bug 修复
### 2.4 性能指标

## 三、下周工作计划
### 3.1 计划任务
### 3.2 风险与问题
### 3.3 需要支持
```

**区别**：
- ✅ 结构化（分类清晰，层次分明）
- ✅ 重点突出（重要进展、风险问题标记）
- ✅ 逻辑连贯（工作进展 → 数据统计 → 下周计划）

### 3. 重点突出，不是"面面俱到"

OpenClaw 会自动识别和标记重要信息：

**📌 重要进展**
- 核心功能完成（代码量大、提交频繁）
- 性能优化（响应时间降低、错误率降低）
- 里程碑任务（关联 Jira 里程碑）

**⚠️ 风险问题**
- 任务延期（状态未更新）
- Bug 未修复（高优先级 Bug）
- 性能异常（指标下降）

**💡 优化建议**
- 代码重复（建议重构）
- 技术债务（建议清理）
- 性能瓶颈（建议优化）

## 避坑指南：这些坑我都踩过

### 坑1: 数据源配置错误

**问题**：Git 仓库路径写错，导致无法获取提交记录
**解决**：
```yaml
# 错误示例
git:
  repos:
    - path: ~/project/backend  # 少了 s

# 正确示例
git:
  repos:
    - path: ~/projects/backend
```

**建议**：先手动测试数据源，确保能正常获取数据

---

### 坑2: 时间范围设置错误

**问题**：使用 "week" 时，有些系统默认是周一到周日，但你的周报周期可能是周二到周一
**解决**：
```yaml
# 明确指定时间范围
time_range:
  start: "2026-03-25 00:00:00"  # 周二
  end: "2026-03-31 23:59:59"    # 下周一
```

**建议**：根据你的公司/团队的周报周期调整时间范围

---

### 坑3: 数据量过大导致超时

**问题**：一次同步太多数据（比如过去 1 年的 Git 提交），导致 OpenClaw 超时
**解决**：
```yaml
# 分批同步
git:
  time_range: "week"  # 只同步本周的数据
```

**建议**：只同步需要的时间范围，不要一次性同步太多历史数据

---

### 坑4: 敏感信息泄露

**问题**：周报中包含了敏感信息（API Key、密码等）
**解决**：
```yaml
# 配置敏感信息过滤
filters:
  sensitive:
    - pattern: "API_KEY.*=.*"
      replace: "API_KEY=***"
    - pattern: "password.*=.*"
      replace: "password=***"
```

**建议**：配置敏感信息过滤规则，避免泄露

---

### 坑5: 领导不习惯新格式

**问题**：领导习惯了原来的手动周报格式，对自动周报的新格式不适应
**解决**：
```yaml
# 自定义周报模板
template: |
  ## 工作进展
  {{work_progress}}

  ## 数据统计
  {{data_stats}}

  ## 下周计划
  {{next_week_plan}}
```

**建议**：根据领导的偏好调整周报模板，逐步过渡
