## 核心知识点

### 1. RAG 的基本原理

**RAG（Retrieval-Augmented Generation）** 是一种结合信息检索与模型生成的技术，通过引入外部知识库或检索系统，增强生成式模型的知识范围和回答准确性。

**核心思想：**
- 大模型存在知识局限性（训练数据截止时间、私有数据、领域专业知识）
- RAG 通过实时检索外部知识库，将相关信息注入生成过程
- 无需重新训练模型，即可扩展知识范围

**发展历程：**
- 2020年由 Meta（前 Facebook AI Research）提出
- 显著提升了知识密集型 NLP 任务性能
- 成为当前企业级应用的主流方案

---

### 2. RAG 的两大核心环节

根据架构图，RAG 系统分为两个关键环节：

#### **离线构建环节（Offline）**

**目标：** 构建可检索的知识向量库

**主要步骤：**
1. **文档收集**
   - 从多个数据源收集文档（PDF、网页、数据库等）
   - 文档清洗与格式化

2. **分块（Chunking）**
   - 将长文档切分成小块（通常 512-1024 tokens）
   - 保留上下文语义完整性
   - 重叠切分避免信息丢失（重叠率 10%-20%）

3. **向量化（Embedding）**
   - 使用嵌入模型将文本块转换为向量
   - 常用模型：OpenAI text-embedding-3、BGE、M3E 等
   - 向量维度：768-1536 维不等

4. **构建索引**
   - 使用向量数据库（FAISS、Milvus、Pinecone、Chroma 等）
   - 支持高效的相似度搜索
   - 可选：元数据过滤、时间过滤等

#### **在线服务环节（Online）**

**目标：** 实时检索并生成答案

**主要步骤：**
1. **问题理解**
   - 用户提问 → 问题向量化
   - 可选：问题改写、Query Expansion（扩展）

2. **检索（Retrieval）**
   - 在向量库中查找 Top-K 最相关文档块
   - 相似度计算：余弦相似度、欧氏距离等
   - Top-K 通常为 5-10

3. **上下文构建**
   - 将检索到的文档块拼接成上下文
   - 按相关性排序
   - 控制总长度（避免超过模型上下文窗口）

4. **生成（Generation）**
   - 将问题 + 上下文输入大模型
   - Prompt 模板示例：
     ```
     请根据以下参考信息回答问题。如果参考信息中没有答案，请明确说明。

     参考信息：
     {context}

     问题：{question}

     答案：
     ```
   - 大模型基于检索到的上下文生成答案

5. **答案输出**
   - 可添加引用来源（citation）

### 3. RAG vs. 微调的选择

| 对比维度 | RAG | 微调（SFT/LoRA） |
|---------|-----|-----------------|
| **知识更新** | 实时更新，无需重新训练 | 需要重新训练 |
| **私有数据** | 完全支持 | 可能泄露到模型参数 |
| **幻觉问题** | 较低（基于检索内容） | 可能产生幻觉 |
| **开发成本** | 较低 | 较高（需要数据和训练） |
| **推理成本** | 较高（检索 + 生成） | 较低（仅生成） |
| **适用场景** | 知识密集型问答、文档助手 | 特定格式输出、风格对齐 |

**建议：**

**优先使用 RAG 的情况：**
- 需要最新知识（新闻、实时数据）
- 有私有文档（企业知识库、内部文档）
- 需要可追溯的答案来源
- 快速原型验证

**考虑微调的情况：**
- 特定格式输出（JSON、SQL）
- 风格和语调对齐
- 知识需要内化到模型
- 需要降低推理延迟

**RAG + 微调组合：**
- 用 RAG 提供知识
- 用微调对齐格式和风格

### 4. RAG 的关键优化技术

#### **4.1 检索优化**

**混合检索（Hybrid Search）：**
- 结合向量检索 + 关键词检索（BM25）
- 关键词检索：准确匹配专有名词
- 向量检索：理解语义
- 融合方式：加权打分、倒排合并

**重排序（Reranking）：**
- 先粗检索更多文档（Top-50）
- 再用重排序模型精确排序（Top-10）
- 常用模型：BGE-Reranker、Cohere Rerank
- 提升最终质量，但增加延迟

**Query 改写：**
- 原始问题 → 更适合检索的查询
- 方法：
  - 多查询生成：生成多个相关问题
  - HyDE（Hypothetical Document Embeddings）：先生成假设答案，再用答案向量检索

#### **4.2 向量化优化**

**嵌入模型选择：**
- 通用：OpenAI text-embedding-3、Cohere embed
- 中文：BGE-large-zh、M3E
- 多语言：multilingual-e5-large
- 领域特定：法律、医疗、金融领域模型

**分块策略：**
- 固定大小：简单但可能切分不合理
- 语义分块：按段落、句子边界切分
- 递归分块：尝试多种大小
- 父子分块：索引小块，检索返回父块（保留上下文）

#### **4.3 上下文优化**

**上下文长度控制：**
- 避免超过模型上下文窗口
- 按相关性截断
- 只保留最相关的 Top-N 块

**上下文去重：**
- 检索结果可能包含重复信息
- 去重提升效率

**上下文排序：**
- 按相关性排序
- 按时间排序（适合新闻类）

#### **4.4 生成优化**

**Prompt Engineering：**
- 明确指令：要求基于检索内容
- 角色设定：设定回答风格
- 示例：Few-shot 示例引导格式

**引用来源：**
- 在答案中标注引用的文档
- 提升可信度

**拒绝回答：**
- 明确指示：如果检索内容中没有答案，说明不知道
- 避免幻觉

### 5. RAG 的常见框架与工具

#### **向量数据库**

| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| **FAISS** | Meta 开源，高性能 | 本地部署、大规模 |
| **Milvus** | 开源，功能全 | 企业级应用 |
| **Pinecone** | 托管服务 | 快速原型、中小规模 |
| **Chroma** | 轻量级，易用 | 个人项目、学习 |
| **Qdrant** | 开源，Rust 实现 | 高性能需求 |
| **Weaviate** | 支持多模态 | 图像 + 文本 |

#### **RAG 框架**

**LangChain：**
- 最流行的 LLM 应用框架
- 提供完整的 RAG 链路
- 支持多种向量数据库

**LlamaIndex：**
- 专注于 RAG 的框架
- 数据索引和检索优化
- 支持高级索引（树、知识图谱）

**Haystack：**
- deepset 开源
- 企业级搜索 + RAG

**Semantic Kernel：**
- Microsoft 开发
- 与 Azure 集成好

#### **快速上手示例（LangChain）**

```python
from langchain.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
from langchain.chains import RetrievalQA
from langchain.chat_models import ChatOpenAI

# 1. 加载文档
loader = TextLoader("document.txt")
documents = loader.load()

# 2. 文档分块
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=200
)
splits = text_splitter.split_documents(documents)

# 3. 构建向量数据库
vectorstore = Chroma.from_documents(
    documents=splits,
    embedding=OpenAIEmbeddings()
)

# 4. 创建 RAG 链
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})
qa_chain = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(temperature=0),
    chain_type="stuff",
    retriever=retriever,
    return_source_documents=True
)

# 5. 查询
query = "什么是 RAG？"
result = qa_chain({"query": query})

print(f"答案：{result['result']}")
print(f"来源：{result['source_documents']}")
```

## 实践建议

### 1. 从简单开始
- 先用基础 RAG 验证可行性
- 不要一开始就用所有优化技术

### 2. 质量评估
- 建立 RAG 评估体系
- 评估维度：
  - 检索准确率（Retrieval Accuracy）
  - 答案相关性（Relevance）
  - 事实正确性（Faithfulness）
  - 拒绝率（当无法回答时是否拒绝）

### 3. 持续迭代
- 收集用户反馈
- 分析检索失败案例
- 优化分块策略、嵌入模型

### 4. 成本优化
- 缓存常见查询
- 批量处理
- 选择合适的模型大小

### 5. 安全性
- 文档权限控制
- 敏感信息过滤
- 访问日志审计

## 常见问题与解决

### Q1: 检索到无关文档？
**原因：** 嵌入模型不匹配、分块不合理、查询模糊
**解决：**
- 更换领域特定的嵌入模型
- 优化分块策略（父子分块、语义分块）
- 使用 Query 改写

### Q2: 答案不基于检索内容？
**原因：** Prompt 不明确、模型产生幻觉
**解决：**
- 优化 Prompt，强调"基于参考信息"
- 添加负向指令："如果参考信息中没有答案，请说明不知道"
- 降低生成温度（temperature=0）

### Q3: 响应慢？
**原因：** 检索 + 生成两步耗时
**解决：**
- 使用更快的向量数据库
- 减少检索数量
- 使用缓存
- 模型量化

### Q4: 上下文超过模型限制？
**原因：** 检索太多文档块
**解决：**
- 减少 Top-K
- 优化分块大小
- 使用长上下文模型（如 Claude 200K、GPT-4-Turbo 128K）
