## 知识点一：KV Cache — 自回归推理的"记忆机制"

### 问题

自回归生成时，第 t 步需要计算 attention(q_t, K_1:t, V_1:t)。如果每步都重新计算所有 token 的 K/V，计算量是 O(t²)，大量重复。

### 原理

KV Cache 将每一步新生成的 Key 和 Value 缓存下来，下一步只需：
1. 计算新 token 的 Q/K/V
2. 新 K/V 追加到缓存
3. Attention 仅对 q_t 和缓存的 K/V 计算

```
# 伪代码示意
# 步骤 t=3，已有缓存 k_cache=[k0,k1,k2], v_cache=[v0,v1,v2]
q3, k3, v3 = linear(x3)           # 仅计算新 token
k_cache.append(k3)                 # 追加缓存
v_cache.append(v3)
output = attention(q3, k_cache, v_cache)  # q3 对全部 KV 做注意力
```

### 显存代价

KV Cache 占用显存 = `2 × num_layers × seq_len × hidden_dim × dtype_bytes`

以 LLaMA-2-70B（FP16）为例，seq_len=4096 时 KV Cache ≈ **32GB**，接近模型权重本身。

> 💡 这也是 GQA/MQA 减少 KV 头数的核心动机——压缩 KV Cache 显存。

## 知识点二：Continuous Batching — 消除"等待空转"

### 问题

传统 Static Batching：一个 batch 中所有序列必须等最长的序列完成后才能释放。短序列空等，GPU 利用率低。

```
Static Batching:
请求A: [生成...生成...生成...] ✅完成  空等空等空等
请求B: [生成..............生成..............] ✅完成
请求C: [生成...✅完成  空等空等空等空等空等空等]
                                  ↑ C完成后仍在等B
```

### 原理

Continuous Batching（也称 In-Flight Batching）：
- 每个请求独立调度，生成完毕立即移出
- 空出的槽位立即填入新请求
- 实现 **iteration-level** 的调度粒度

```
Continuous Batching:
迭代1: [A, B, C]     → 三个请求同时推理一步
迭代2: [A, B, C]     → C 完成，新请求 D 加入
迭代3: [A, B, D]     → A 完成，新请求 E 加入
迭代4: [B, D, E]     → 持续流动，无空转
```

### 关键技术

- **PagedAttention**（vLLM）：借鉴 OS 虚拟内存分页思想，KV Cache 按块分配，非连续存储，消除碎片
- **Preemption**：显存不足时抢占低优先级请求，释放 KV Cache 块
- **Sliding Window**：只保留最近 W 个 token 的 KV，适合长对话场景

### 效果

vLLM 论文报告：相比 Static Batching，吞吐量提升 **2-4x**，P99 延迟降低 **50%+**。

## 知识点三：FlashAttention — IO 感知的精确注意力

### 问题

标准 Attention 的瓶颈不在计算（FLOPs），而在**显存带宽（HBM ↔ SRAM）**：
- Q/K/V 需要从 HBM 加载到 SRAM
- 中间矩阵 S = QK^T（n×n）要写回 HBM 再读回计算 softmax
- 显存访问量 O(N²)，成为 wall-clock 瓶颈

### 核心思想

FlashAttention 通过 **Tiling（分块计算）** 减少 HBM 读写：

1. 将 Q/K/V 分成小块（tile），每块放入 SRAM
2. 在 SRAM 内完成该块的 attention 计算
3. 用 **online softmax** 技巧（逐步归一化）避免需要全部 S 矩阵
4. 只将最终 O 写回 HBM

```
# 标准 Attention:  O(N²) HBM 读写
S = Q @ K.T          # N×N，写回HBM
P = softmax(S)       # 从HBM读取S
O = P @ V            # 写回HBM

# FlashAttention:   O(N) HBM 读写
for each tile (q_block, k_block, v_block):
    s_tile = q_block @ k_block.T    # 在SRAM中
    o_tile = online_softmax_update(o_tile, s_tile, v_block)  # SRAM中
write o_tile to HBM                  # 仅最终结果写回
```

### FlashAttention-2 改进

- 优化线程块分配：减少非矩阵乘法操作
- 针对 GPU Tensor Core 优化：最大化矩阵乘法单元利用率
- 速度比 FlashAttention-1 快 **~2x**

### 效果对比

| 方法 | 显存 | HBM 读写 | 精确性 |
|------|------|---------|--------|
| 标准 Attention | O(N²) | O(N²) | ✅ 精确 |
| FlashAttention | O(N) | O(N²/d_round) | ✅ 精确（数学等价） |
| 稀疏 Attention | O(N) | O(N) | ❌ 近似 |

> FlashAttention 是**精确计算**，不是近似！数学上与标准注意力完全等价。

## 知识点四：推理优化全景图

将推理优化按层次梳理：

| 层次 | 技术 | 效果 |
|------|------|------|
| **服务层** | Continuous Batching、动态批处理 | 吞吐 +2-4x |
| **模型层** | 量化（INT8/INT4）、蒸馏、剪枝 | 模型体积 -50%~75% |
| **算子层** | FlashAttention、Fused Kernels | 延迟 -30%~50% |
| **系统层** | KV Cache、PagedAttention | 显存 -50%，支持更长序列 |
| **硬件层** | Tensor Core、NVLink、多卡并行 | 线性扩展 |

### 生产环境推荐组合

```
推理服务 = vLLM/PagedAttention + FlashAttention-2 + Continuous Batching + (可选) INT8 量化
```

```python
# PyTorch 中自动启用 FlashAttention
import torch
# SDPA 会自动选择 FlashAttention / MemEfficient / Math 后端
output = torch.nn.functional.scaled_dot_product_attention(q, k, v)
```

## 关键洞察

> **推理优化的本质是减少"等待"**——KV Cache 减少重复计算的等待，Continuous Batching 减少批次内空转的等待，FlashAttention 减少数据搬运的等待。三者互补，协同加速。
