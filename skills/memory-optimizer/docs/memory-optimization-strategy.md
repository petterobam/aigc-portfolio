# 记忆优化策略

> 本文档描述记忆优化器的核心优化策略，包括评分体系、归档规则、去重逻辑和清理机制。
> 这是系统决策的依据文档，修改策略前必须先更新本文档。

---

## 一、优化目标

记忆系统的核心矛盾：
- 记忆越多，检索越慢，噪音越大
- 删得太激进，有价值的记忆永久丢失
- 删得太保守，系统退化为垃圾堆

**优化目标**：在保留所有有价值记忆的前提下，最小化无效记忆的存量。

**衡量标准**：
- 误删率 < 1%（有价值记忆被误删的比例）
- 误归档率 < 5%（活跃记忆被归档的比例）
- 垃圾记忆占比 < 10%（无价值记忆 / 总记忆数）
- 平均重要性评分 > 2.5（整体质量基线）

## 二、重要性评分体系

### 评分公式

```
importance = (访问频率 × 0.4)
           + (时效性   × 0.3)
           + (内容质量 × 0.2)
           + (关联度   × 0.1)
```

### 各维度说明

#### 访问频率（权重 0.4）

```javascript
accessFrequency = access_count / Math.max(1, age_days)
```

- 含义：平均每天被访问多少次
- 高分特征：频繁被引用、经常被查询
- 低分特征：创建后从未被访问过

#### 时效性（权重 0.3）

```javascript
recency = 1 / Math.max(1, days_since_last_access)
```

- 含义：距上次访问越近，分越高
- 高分特征：最近 7 天内被访问
- 低分特征：超过 30 天未被访问

#### 内容质量（权重 0.2）

```javascript
contentQuality = Math.min(1, content_length / 1000)
```

- 含义：内容越完整，分越高（上限 1000 字）
- 高分特征：内容详细、有标签、结构清晰
- 低分特征：内容极短、无标签、内容碎片化

> 未来改进：加入标签完整性评分（有标签 +0.2，多个标签 +0.1）

#### 关联度（权重 0.1）

```javascript
referenceCount = memory.reference_count || 0
```

- 含义：被其他记忆引用的次数
- 高分特征：是多条记忆的"基础知识"
- 低分特征：孤立记忆，无任何关联

### 评分区间解读

| 分值区间 | 含义 | 处理建议 |
|---------|------|---------|
| 0 ~ 1 | 低价值，可能是垃圾记忆 | 候选清理 |
| 1 ~ 2 | 价值偏低，可考虑归档 | 候选归档 |
| 2 ~ 3 | 正常记忆，保留 | 维持现状 |
| 3 ~ 4 | 高价值记忆，重点保留 | 保护 |
| 4 ~ 5 | 核心记忆，永久保留 | 建议加 protected 标签 |

## 三、去重策略

### 第一层：精确去重（内容哈希）

```javascript
hash = MD5(memory.content)
// 哈希完全相同 → 内容完全相同 → 合并
```

**合并规则**：
- 保留创建时间较早的（原始记忆）
- 保留访问次数较多的（更活跃）
- 若两条都有标签，取并集
- 被合并的记忆归档（不删除），标注 `merged-into: {id}`

**适用场景**：同一条知识被重复写入了两次

### 第二层：标题去重

```javascript
// 标题完全相同 → 大概率是重复写入
// 保留最新的，归档旧的
```

**注意**：标题相同但内容不同时，视为更新版本而非重复，**不做合并**。

### 第三层：语义去重（规划中，未实现）

```javascript
similarity = cosineSimilarity(embedding_A, embedding_B)
// similarity > 0.95 → 语义重复 → 人工确认后合并
```

**实现前置条件**：
- 需要向量化模型（OpenAI Embeddings 或本地模型）
- 需要向量存储（SQLite FTS5 或 LanceDB）
- 相似度阈值建议设为 0.95，避免误合并

**当前状态**：待开发，见 tasks/task-list.md #2

## 四、归档规则

归档 ≠ 删除。归档的记忆仍然存在于数据库，只是从活跃列表中移除，不参与日常检索。

### 标准归档条件（同时满足）

```
days_since_last_access > 30 天
AND importance < 2
AND tags 不包含 'protected'
```

### 加速归档条件（同时满足）

```
days_since_last_access > 60 天
AND importance < 3
AND tags 不包含 'protected'
```

### 永不归档的情况

- 包含 `protected` 标签
- importance >= 4
- 被其他活跃记忆引用（reference_count > 0）
- category 为 `system` 或 `config`

### 归档后的操作

```sql
UPDATE metadata SET category = 'archived' WHERE id = ?
```

归档记忆不会出现在日常查询结果中，但可以通过以下方式找回：

```sql
SELECT * FROM metadata WHERE category = 'archived' ORDER BY importance DESC
```

## 五、清理规则

清理 = 永久删除，不可恢复。**必须极度谨慎。**

### 当前配置：deleteArchived = false

默认情况下，优化脚本**不执行任何删除操作**。所有低质量记忆只做归档处理。

### 启用删除的条件（需手动开启）

```javascript
// 在 CONFIG 中设置
deleteArchived: true
```

启用后，以下两类记忆会被删除：

**低质量记忆**：
```
content_length < 50 字符
AND importance < 1
AND tags 为空
AND days_since_creation > 7 天
```

**长期归档记忆**（可选，默认不启用）：
```
category = 'archived'
AND days_since_archived > 90 天
AND importance < 1
```

### 删除前的保护措施

1. 删除前自动创建数据库备份：`memory-backup-{timestamp}.db`
2. 删除操作写入删除日志：`logs/deletion-log.md`
3. 每次运行最多删除 20 条（防止批量误删）
4. 任何带 `protected` 标签的记忆，无论如何都不删除

## 六、受保护记忆

标记 `protected` 标签的记忆完全免疫所有优化操作：

```javascript
if (memory.tags.includes('protected')) {
  stats.protected++
  return  // 直接跳过，不做任何处理
}
```

**哪些记忆应该标记 protected**：
- 系统配置类记忆（重建成本高）
- 独特经验总结（不可再现的洞见）
- 关键决策记录（为什么做了某个决定）
- 长期积累的方法论（反复验证的知识）

**如何标记**：

```sql
-- 通过 SQLite 直接标记
UPDATE metadata
SET tags = json_insert(tags, '$[#]', 'protected')
WHERE id = {id};
```

## 七、优化频率与时机

### 定时执行（每 30 分钟）

```cron
*/30 * * * * cd ~/.openclaw/workspace && node skills/memory-optimizer/scripts/optimize.js
```

适合：常规维护，低强度优化

### 心跳触发执行

在心跳时刻手动触发，执行完整优化并分析报告。

适合：发现异常后的深度清理、调整参数后的验证

### 不适合执行优化的时机

- 正在大批量写入新记忆时（避免误归档刚写入的记忆）
- 数据库刚刚恢复备份后（先检查数据完整性）
- 修改了 CONFIG 参数但还没有验证时（先小范围测试）

## 九、已知局限与未来改进方向

| 局限 | 影响 | 改进方向 | 优先级 |
|------|------|---------|-------|
| 只有哈希去重，无法识别语义重复 | 语义重复记忆会留存 | 引入向量相似度 | P1 |
| 内容质量评分过于简单（只看长度） | 短但高价值的记忆会被低估 | 加入标签完整性、结构评分 | P2 |
| 无自动标签提取 | 缺标签的记忆评分偏低 | 关键词自动提取 | P2 |
| 归档后无自动通知 | 归档了有价值记忆不易发现 | 归档摘要推送到 heartbeat | P2 |
| 无增量优化，每次全量扫描 | 记忆量大时性能下降 | 只扫描最近更新的记忆 | P3 |
