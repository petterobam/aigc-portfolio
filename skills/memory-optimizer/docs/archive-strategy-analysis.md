## 问题分析

### 核心问题：归档策略永远不会触发

**原因**：

1. **访问追踪系统定期更新**
   - 访问追踪系统（access-tracker.js）定期运行（每次心跳）
   - 每次运行都会更新 `last_accessed` 为当前时间
   - `days_since_last_access` 重新从 0 开始累积

2. **归档条件无法满足**
   - `days_since_last_access >= 30` 永远无法满足
   - 因为 `last_accessed` 每次心跳都会更新
   - 归档逻辑形同虚设

3. **访问追踪系统的目的冲突**
   - 访问追踪系统用于计算启发式访问频率，用于评分系统
   - 归档策略使用真实的 `days_since_last_access`，用于判断记忆是否过期
   - 两个系统对 `last_accessed` 的使用方式冲突

## 根因分析

### 问题 1：访问追踪系统没有定期运行

**现象**：
- 访问追踪系统只在 2026-03-21 23:08:07 运行过一次
- 之后就没有再运行过（直到本次心跳）
- `days_since_last_access` 一直保持在 2 天

**影响**：
- 归档策略使用真实的 `days_since_last_access`，但这个值没有更新
- 访问追踪系统用于评分系统，但访问频率数据过时

**解决方案**：
- 方案 1：在 cron 中配置定期任务（每天运行一次）
- 方案 2：在优化脚本（optimize.js）中集成访问追踪逻辑（每次运行都更新）
- 方案 3：分离启发式访问追踪和真实访问追踪

### 问题 2：归档策略与访问追踪系统冲突

**现象**：
- 访问追踪系统定期更新 `last_accessed`
- 归档策略使用 `days_since_last_access` 判断是否归档
- 两个系统对 `last_accessed` 的使用方式冲突

**影响**：
- 如果访问追踪系统定期运行，`days_since_last_access` 永远不会达到 30 天
- 归档策略永远不会触发

**解决方案**：
- 方案 1：分离 `last_accessed`（真实访问）和 `heuristic_last_accessed`（启发式访问）
- 方案 2：归档策略不再使用 `days_since_last_access`，改用其他指标
- 方案 3：减少访问追踪频率（比如每周运行一次）

### 问题 3：归档条件过于单一

**现象**：
- 当前归档条件：`days_since_last_access >= 30 && importance < 1.5`
- 只考虑了"未访问天数"和"重要性评分"两个维度
- 没有考虑访问频率、创建时间、内容长度等因素

**影响**：
- 一些低价值的记忆（如临时笔记、测试记录）长期占据记忆库
- 记忆库逐渐膨胀，影响检索效率和评分准确性

**解决方案**：
- 方案 1：增加更多归档条件（访问频率、创建时间、内容长度）
- 方案 2：调整归档参数（archiveAfterDays, minImportance）
- 方案 3：引入机器学习模型，自动预测哪些记忆应该归档

## 改进方案

### 方案 1：分离真实访问和启发式访问（推荐）

**设计思路**：

1. **分离 `last_accessed` 字段**：
   - `last_accessed`：记录真实的用户访问（用于归档策略）
   - `heuristic_last_accessed`：记录启发式访问追踪（用于评分系统）

2. **修改 metadata 表结构**：
   ```sql
   ALTER TABLE metadata ADD COLUMN heuristic_last_accessed DATETIME;
   ALTER TABLE metadata ADD COLUMN heuristic_access_count INTEGER DEFAULT 0;
   ```

3. **修改访问追踪系统**：
   - access-tracker.js 更新 `heuristic_last_accessed` 和 `heuristic_access_count`
   - 不再更新 `last_accessed` 和 `access_count`

4. **修改归档策略**：
   - 仍然使用 `days_since_last_access`（基于真实访问）
   - 不再受启发式访问追踪影响

**优势**：
- ✅ 归档策略和评分系统解耦，互不干扰
- ✅ 访问追踪系统可以定期运行，不影响归档逻辑
- ✅ 真实访问数据用于归档，启发式访问数据用于评分
- ✅ 逻辑清晰，易于理解和维护

**劣势**：
- ❌ 需要修改数据库 Schema（增加新字段）
- ❌ 需要修改多个脚本（access-tracker.js, optimize.js）
- ❌ 需要处理现有数据的迁移

**预估工作量**：2-3 小时

---

### 方案 2：调整归档参数（快速修复）

**设计思路**：

1. **调整归档条件**：
   - `archiveAfterDays`：30 天 → 14 天
   - `minImportance`：1.5 → 1.2
   - 或者增加新的归档条件（如访问频率过低）

2. **配置定期任务**：
   - 访问追踪系统每周运行一次（而不是每次心跳）
   - 或者完全禁用自动访问追踪，改为手动运行

**优势**：
- ✅ 无需修改数据库 Schema
- ✅ 修改量小，快速实施
- ✅ 可以立即见效

**劣势**：
- ❌ 访问追踪系统不能定期运行，影响评分准确性
- ❌ 归档条件仍然可能与访问追踪系统冲突
- ❌ 不是根本解决方案，只是缓解问题

**预估工作量**：0.5-1 小时

---

### 方案 3：改进归档策略（综合方案）

**设计思路**：

1. **增加更多归档条件**：
   - 访问频率过低（`access_count < threshold`）
   - 创建时间过久（`created_at < threshold`）
   - 内容长度过短（`content_length < threshold`）
   - 评分持续下降（连续 N 次心跳评分下降）

2. **动态调整归档参数**：
   - 根据记忆库大小自动调整 `archiveAfterDays`
   - 根据评分分布自动调整 `minImportance`
   - 根据归档率自动调整归档条件

3. **引入归档候选机制**：
   - 每次心跳生成"归档候选列表"
   - 用户可以手动确认或批量归档
   - 避免误归档高价值记忆

**优势**：
- ✅ 归档策略更灵活，适应不同场景
- ✅ 自动调整参数，减少人工干预
- ✅ 归档候选机制，避免误归档

**劣势**：
- ❌ 归档策略更复杂，难以理解和维护
- ❌ 可能需要机器学习模型，增加系统复杂度
- ❌ 需要更多测试和验证

**预估工作量**：1-2 天
