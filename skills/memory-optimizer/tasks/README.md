# 任务管理 - 记忆优化器

> 任务列表是心跳系统的行动轨道。每次心跳至少推进一个任务，或新增一个任务。
> 没有更新任务列表的心跳，等于在原地踏步。

---

## 目录结构

```
tasks/
├── README.md        本文件（任务管理规范）
├── task-list.md     所有任务（P0~P3，唯一事实来源）
└── completed/       已完成任务归档（可选，积累后迁移）
```

---

## 优先级定义

| 优先级 | 含义 | 时间框架 | 处理原则 |
|-------|------|---------|---------|
| P0 | 正在进行，阻塞其他工作 | 立即完成 | 每次心跳必须推进 |
| P1 | 本周必须完成 | 7 天内 | 优先于新功能开发 |
| P2 | 本月完成 | 30 天内 | 有空档时推进 |
| P3 | 规划中，低优先级 | 下月及以后 | 不紧急，但要记录 |

---

## 任务状态标记

| 标记 | 含义 |
|-----|------|
| `[ ]` | 待开始 |
| `[x]` | 已完成 |
| `[~]` | 进行中（已开始但未完成） |
| `[-]` | 已取消（注明原因） |
| `[?]` | 待评估（不确定是否要做） |

---

## 与 HEARTBEAT 的流转关系

```
心跳开始
    ↓
查看 state/current-state.md（了解系统现状）
    ↓
打开 task-list.md（找到最高优先级未完成任务）
    ↓
执行任务（运行脚本 / 写代码 / 调研文档）
    ↓
更新任务状态（勾选完成 / 更新进度）
    ↓
更新 logs/latest.md 或 state/current-state.md
    ↓
心跳结束
```

**关键原则**：每次心跳结束时，task-list.md 里至少有一行状态发生了变化。

## 任务更新规范

### 完成一个任务时
1. 在 task-list.md 中将 `[ ]` 改为 `[x]`
2. 在任务下方补一行简短结果（实际完成了什么）
3. 更新底部「任务统计」的数字
4. 在「任务更新记录」追加一条记录

### 新增一个任务时
1. 确定优先级（P0~P3）
2. 写清楚「目标」「子任务」「预期成果」「预估时间」
3. 更新「任务统计」总数

### 取消一个任务时
1. 将 `[ ]` 改为 `[-]`
2. 在任务下补注取消原因（一句话即可）
3. 不要删除，留作记录

### 调整优先级时
1. 移动任务到对应优先级区块
2. 在「任务更新记录」注明调整原因

---

## 何时创建新任务

以下情况应立即在 task-list.md 中新增任务：

- 运行 optimize.js 发现了脚本 bug 或异常
- 在 state/current-state.md 中记录了新的已知问题
- 调研过程中发现了值得实现的技术方向
- 收到来自其他 skill（heartbeat/sqlite-memory 等）的改进建议

---

## 任务完成后的归档

当 task-list.md 中已完成任务（`[x]`）累积超过 10 条时：
1. 将这些已完成任务剪切到 `completed/YYYYMM-completed.md`
2. 保留任务标题和完成日期，方便回溯
3. 更新 task-list.md 底部的统计数字
