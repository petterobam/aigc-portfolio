## 归档策略分析

### 当前归档策略

```javascript
const shouldArchive = (memory.importance_score < OPTIMIZE_CONFIG.minImportance) &&
                     (memory.days_since_last_access >= OPTIMIZE_CONFIG.archiveAfterDays);
```

**参数**:
- `minImportance`: 1.5（重要性低于此值的记忆才可能被归档）
- `archiveAfterDays`: 30（超过 30 天未访问才归档）

### 归档策略问题

| 问题 | 严重程度 | 说明 |
|------|---------|------|
| 归档阈值过高 | 中 | 30 天未访问才归档，对于短期记忆过于宽松 |
- 重要性阈值合理 | 低 | minImportance = 1.5 能够区分高价值记忆 |
- 时间条件单一 | 中 | 只考虑"未访问天数"，未考虑访问频率趋势 |
- 归档条件过于严格 | 高 | 两个条件必须同时满足（AND 逻辑），降低了归档概率 |

**影响**:
- ⚠️ 低价值记忆长期滞留在记忆库中（operation: 0.8, task: 1.0）
- ⚠️ 记忆库可能积累过多历史数据，影响检索效率
- ⚠️ 归档策略在记忆库建立初期（前 1 个月）完全失效

## 改进建议

### 短期方案（立即实施）

#### 1. 降低归档时间阈值

**推荐方案**: 将 `archiveAfterDays` 从 30 天降低到 14 天

**理由**:
- 14 天是 2 周，足够观察记忆的访问模式
- 低价值记忆在 2 周内如果没有被访问，归档风险较低
- 14 天归档后，仍然可以通过查询归档记忆找回

**风险评估**:
- ⚠️ 可能增加误归档风险（归档了还有价值的记忆）
- ✅ 但归档记忆不会被删除，可以通过查询找回
- ✅ 如果发现误归档，可以撤销归档（将 archived 改为其他分类）

**实施步骤**:
1. 修改 `scripts/config.js` 中的 `archiveAfterDays: 30` → `archiveAfterDays: 14`
2. 运行优化脚本，验证归档逻辑
3. 观察 1-2 周，评估归档效果
4. 如果归档过多，可以调整到 21 天

#### 2. 增加访问频率归档条件

**推荐方案**: 添加访问频率归档条件

```javascript
const lowAccessFrequency = (memory.access_count / memory.age_days) < 0.5; // 平均每天访问次数 < 0.5

const shouldArchive = lowAccessFrequency &&
                     (memory.importance_score < OPTIMIZE_CONFIG.minImportance) &&
                     (memory.age_days >= 7); // 至少 7 天后才归档
```

**理由**:
- 如果记忆的平均访问频率 < 0.5（即每 2 天才访问一次），说明访问频率很低
- 结合重要性评分（< 1.5）和最小年龄（>= 7 天），可以更智能地归档
- 7 天归档条件比 30 天更激进，但访问频率条件提供了额外的安全阀

**实施步骤**:
1. 修改 `scripts/optimize.js` 中的归档逻辑
2. 添加访问频率计算和归档条件
3. 运行优化脚本，验证归档效果
4. 观察归档数量，评估是否需要调整阈值

#### 3. 实现软归档（优先级：低）

**推荐方案**: 实现"软归档"机制，而非硬删除

**软归档定义**:
- 软归档：将记忆标记为 archived，但仍可在查询结果中显示（带"已归档"标签）
- 硬删除：彻底删除记忆，无法恢复

**实施步骤**:
1. 修改归档逻辑，只标记 archived，不删除数据
2. 修改查询逻辑，默认不显示 archived 记忆，但可以通过参数 `includeArchived=true` 查询
3. 添加"撤销归档"功能，将 archived 记忆恢复为活跃状态
4. 添加"批量操作"功能，批量归档、批量恢复

**优势**:
- ✅ 降低误归档风险（可以随时撤销归档）
- ✅ 保留历史数据，便于分析和审计
- ✅ 用户体验更好（可以看到被归档的记忆）

### 中期方案（1-2 周内实施）

#### 4. 实现归档预测模型

**推荐方案**: 基于访问频率趋势，预测记忆的未来访问概率

**预测逻辑**:
- 计算记忆的访问频率趋势（最近 7 天 vs 前 7 天）
- 如果访问频率持续下降（趋势为负），预测未来访问概率低
- 结合重要性评分，预测记忆是否应该归档

**实施步骤**:
1. 扩展 access_log 表，记录每次访问的时间戳
2. 实现访问频率趋势计算函数
3. 实现未来访问概率预测函数
4. 将预测结果集成到归档逻辑

#### 5. 实现归档效果评估

**推荐方案**: 定期评估归档效果，识别误归档和遗漏归档

**评估指标**:
- 归档后的记忆是否被重新访问（误归档）
- 未归档的记忆是否长期未被访问（遗漏归档）
- 归档前后的记忆库检索效率变化

**实施步骤**:
1. 记录每次归档的详细信息（归档时间、归档原因）
2. 定期查询归档记忆的访问记录
3. 生成归档效果报告
4. 根据报告调整归档参数

#### 6. 实现智能归档系统

**推荐方案**: 基于机器学习，自动优化归档策略

**技术方案**:
- 使用历史归档数据训练模型（Logistic Regression / Random Forest）
