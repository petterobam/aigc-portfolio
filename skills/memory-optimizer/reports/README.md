# 优化报告归档

> 本目录存放每次运行 optimize.js 生成的结构化报告（JSON）和人工可读摘要（Markdown）。
> 报告是优化系统的"黑匣子"，没有报告就无法追溯决策依据。

---

## 目录结构

```
reports/
├── README.md                        本文件（报告归档说明）
├── latest-report.json               最新一次优化报告（机器可读）
└── YYYYMMDD-HHMMSS-report.md        历史报告摘要（人工可读，按日期归档）
```

---

## 报告类型说明

### latest-report.json

由 `optimize.js` 自动生成并覆盖，始终反映最近一次运行的结果。

路径：`data/memory-optimization-report.json`（脚本默认路径），建议同步复制一份到本目录。

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| timestamp | string | 执行时间（ISO 8601） |
| config | object | 本次生效的配置参数 |
| stats.totalProcessed | number | 扫描的记忆总数 |
| stats.archived | number | 本次归档的记忆数 |
| stats.deleted | number | 本次删除的记忆数 |
| stats.protected | number | 跳过的受保护记忆数 |
| stats.duplicate | number | 发现的重复记忆数 |
| stats.importanceUpdated | number | 重要性评分更新数 |
| summary.optimizationRate | string | 优化率（归档+删除 / 总数） |
| recommendations | array | 系统给出的改进建议 |

### 历史报告摘要（Markdown）

每次心跳或人工复盘时，将 JSON 报告解读后写成 Markdown 摘要，便于趋势分析。

命名规范：`YYYYMMDD-HHMMSS-report.md`

---

## 报告归档流程

```
运行 optimize.js
    ↓
自动生成 data/memory-optimization-report.json
    ↓
将 JSON 复制到 reports/latest-report.json（覆盖）
    ↓
将旧的 latest-report.json 重命名为 YYYYMMDD-HHMMSS-report.json（存档）
    ↓
更新 logs/latest.md 中的执行摘要
    ↓
更新 state/current-state.md 中的最新数据
```

---

## 趋势分析指引

积累 3 次以上报告后，每次心跳可做以下分析：

- 记忆总量是否在增长？增长是否合理？
- 归档量趋势：持续增高说明活跃记忆在减少，需关注
- 重复记忆数：若居高不下，说明去重规则需要收紧
- 平均重要性：长期下降说明记忆质量在退化
- 优化率：过高（>20%）可能意味着规则过于激进

## 注意事项

- JSON 报告不要手动修改，保持原始输出
- Markdown 摘要是人工解读，可以加入观察和判断
- 报告不要删除，趋势价值高于单次价值
- 如果某次运行异常（脚本报错），也要在此记录，标注"异常运行"
