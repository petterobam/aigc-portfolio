### 2.2 ROI 评估框架

如何判断一个 AI 应用是否值得做？使用这个 ROI 评估框架：

#### 成本构成

| 成本项 | 说明 | 计算方法 |
|--------|------|---------|
| API 调用费用 | GPT、Claude 等模型调用费用 | Token 数量 × 单价 |
| 服务器成本 | 应用部署、数据库、缓存 | 云服务器费用 |
| 人力成本 | 开发、测试、运维 | 人力 × 时间 |
| 维护成本 | Bug 修复、功能迭代 | 常年预算 |

#### 收益量化

| 收益项 | 说明 | 计算方法 |
|--------|------|---------|
| 时间节省 | AI 替代人工节省的时间 | 节省时间 × 时薪 |
| 效率提升 | 单位时间产出增加 | 增加产出 × 单价 |
| 收入增长 | 转化率、客户数增长 | 增量收入 |
| 成本降低 | 减少人力、资源投入 | 降低成本 |

#### ROI 计算公式

```
ROI = (收益 - 成本) / 成本 × 100%

示例：
- 成本：¥10,000/月（API ¥3,000 + 服务器 ¥2,000 + 人力 ¥5,000）
- 收益：¥42,000/月（时间节省 ¥30,000 + 效率提升 ¥12,000）
- ROI = (42,000 - 10,000) / 10,000 × 100% = 320%
```

#### 决策阈值

| ROI 范围 | 决策建议 |
|---------|---------|
| < 50% | 不推荐（成本高于收益）|
| 50%-100% | 谨慎考虑（收益有限）|
| 100%-200% | 值得尝试（有明显收益）|
| > 200% | 强烈推荐（收益显著）|

## 阶段二：技术选型与架构设计

### 3.1 模型选择指南

#### 主流模型对比

| 模型 | 优势 | 劣势 | 成本 | 适用场景 |
|------|------|------|------|---------|
| GPT-4 | 推理能力强、长文本 | 成本高、速度慢 | $0.03/1K tokens (输入)<br>$0.06/1K tokens (输出) | 复杂推理、代码生成 |
| Claude | 长文本处理、安全 | 可用性不稳定 | 按用量计费 | 文档分析、长对话 |
| GPT-3.5 | 性价比高 | 推理能力较弱 | $0.002/1K tokens (输入)<br>$0.002/1K tokens (输出) | 日常任务、简单问答 |
| 开源模型 | 成本可控、隐私保护 | 效果不稳定 | 服务器成本 + 部署成本 | 成本敏感、隐私要求高 |

#### 选择决策树

```
是否涉及敏感数据？
├─ 是 → 开源模型（Llama 2、ChatGLM）
└─ 否 → 是否需要强推理能力？
        ├─ 是 → GPT-4
        └─ 否 → GPT-3.5-turbo（性价比高）
```

#### 实际案例

**场景 1：企业内部知识库问答**
- 涉及敏感数据 ✅ → 选择开源模型（ChatGLM-6B）
- 成本：¥0（本地部署，服务器成本 ¥2,000/月）

**场景 2：电商商品描述生成**
- 不涉及敏感数据 ✅ → 不需要强推理能力 ✅ → GPT-3.5-turbo
- 成本：¥0.002/1K tokens，单商品 ¥0.03

**场景 3：代码审查与优化**
- 涉及代码（敏感？）✅ → 代码审查建议开源模型，代码生成建议 GPT-4
- 成本：GPT-4（代码生成）+ 开源模型（代码审查）

**核心原则**：

#### Step 2：API 接口开发

**接口设计原则**：
- 标准化输入输出（便于扩展和集成）
- 异步处理（避免阻塞）
- 错误处理（提供清晰的错误信息）
- 限流保护（防止滥用）

**示例：商品描述生成接口**

```python
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import openai
from typing import Optional

app = FastAPI()

# 数据模型
class ProductDescRequest(BaseModel):
    product_name: str
    keywords: list[str]
    length: Optional[int] = 200  # 默认 200 字

class ProductDescResponse(BaseModel):
    description: str
    tokens_used: int
    cost: float

@app.post("/api/generate-product-desc", response_model=ProductDescResponse)
async def generate_product_desc(request: ProductDescRequest):
    try:
        # 调用 GPT 生成描述
        response = await openai.ChatCompletion.acreate(
            model="gpt-3.5-turbo",
            messages=[
                {"role": "system", "content": "你是一位专业的电商文案..."},
                {"role": "user", "content": f"商品名称：{request.product_name}\n关键词：{', '.join(request.keywords)}"}
            ],
            max_tokens=request.length * 2  # 1 个汉字 ≈ 2 tokens
        )

        description = response.choices[0].message.content
        tokens_used = response.usage.total_tokens
        cost = tokens_used * 0.000002  # GPT-3.5 单价

        return ProductDescResponse(
            description=description,
            tokens_used=tokens_used,
            cost=cost
        )

    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))
```

#### Step 3：缓存层集成

**缓存策略**：
- 缓存键：`product_desc:{product_name}:{keywords_hash}`
- TTL：72 小时（商品信息变化不频繁）
- 缓存更新：手动刷新 + 自动过期

**代码示例**：

```python
import redis
import hashlib

redis_client = redis.Redis(host='localhost', port=6379, db=0)

def generate_product_desc_with_cache(product_name: str, keywords: list[str]):
    # 生成缓存键
    keywords_str = ','.join(sorted(keywords))
    cache_key = f"product_desc:{product_name}:{hashlib.md5(keywords_str.encode()).hexdigest()}"

    # 尝试从缓存获取
    cached = redis_client.get(cache_key)
    if cached:
        return cached.decode('utf-8')

    # 缓存未命中，调用 API
    description = await generate_product_desc(product_name, keywords)

    # 写入缓存（TTL 72 小时）
    redis_client.setex(cache_key, 72 * 3600, description)

    return description
```

### 4.2 性能优化技巧

#### 优化 1：Prompt 压缩

**问题**：冗长的 Prompt 导致 Token 消耗高

**解决方案**：
- 移除不必要的上下文
- 使用变量代替重复文本
- 简化指令（保留核心要求）

**案例**：
```
优化前：600 tokens/请求，成本 ¥0.012
优化后：400 tokens/请求，成本 ¥0.008
收益：成本降低 33%
```

#### 优化 2：批量处理

**问题**：逐个处理请求，效率低

**解决方案**：
- 合并相似请求
- 使用异步并发处理
- 批量调用 API

**代码示例**：
```python
import asyncio

async def batch_generate(product_list: list[dict]):
    tasks = [generate_product_desc(p['name'], p['keywords']) for p in product_list]
    results = await asyncio.gather(*tasks)
    return results
```

**收益**：吞吐量提升 2-3 倍

#### 优化 3：异步处理

**问题**：同步处理阻塞请求

**解决方案**：
- 使用任务队列（Celery、RQ）
- 异步返回任务 ID
- WebSocket 推送结果

**架构**：
```
用户请求 → API 网关 → 任务队列 → Worker → Redis 缓存
                              ↓
                        WebSocket 推送
```

#### 优化 4：模型路由

**问题**：所有请求都用 GPT-4，成本高

**解决方案**：
- 简单任务用 GPT-3.5
- 复杂任务用 GPT-4
- 根据任务难度动态选择

**策略**：
```python
def select_model(task_complexity: str):
    if task_complexity == "low":
        return "gpt-3.5-turbo"
    elif task_complexity == "high":
        return "gpt-4"
    else:
        return "gpt-3.5-turbo"  # 默认
```

#### 策略 3：Token 计费监控

**目标**：设置预算告警，避免超支

**方法**：
- 实时统计 Token 使用量
- 设置阈值（预算的 80%、90%、100%）
- 超额自动告警（邮件、短信）

**代码示例**：
```python
def check_budget():
    current_cost = get_current_cost()
    budget = get_monthly_budget()

    if current_cost > budget * 0.8:
        send_alert("API 费用已超过预算 80%", level="warning")
    elif current_cost > budget * 0.9:
        send_alert("API 费用已超过预算 90%", level="critical")
    elif current_cost > budget:
        send_alert("API 费用已超过预算 100%！", level="emergency")
```

## 常见问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 响应速度慢 | API 延迟 + 无缓存 | 添加缓存层 + 异步处理 |
| 成本过高 | 模型选择不当 + 无缓存 | 模型分级 + 缓存优化 |
| 效果不稳定 | Prompt 设计不当 | Prompt 工程优化 + Fine-tuning |
| 数据安全风险 | 敏感数据泄露 | 数据脱敏 + 本地部署 |
| 扩展性差 | 架构设计不合理 | 模块化 + 水平扩展 |

## 实战案例：电商 AI 描述生成系统

### 项目背景

某电商平台，日均上新 10,000+ 商品：

- **原有流程**：人工撰写商品描述，耗时 2-3 小时/商品
- **成本**：10 人团队，月人力成本 ¥500,000
- **目标**：用 AI 自动生成商品描述，降低成本，提升效率

### 技术方案

**模型选择**：
- GPT-3.5-turbo（性价比高，单次成本 ¥0.03）

**Prompt 设计**：
```
你是一位专业的电商文案，擅长撰写吸引人的商品描述。

任务：根据商品名称和关键词，生成 200 字的商品描述。

要求：
1. 突出核心卖点（基于关键词）
2. 语言简洁有力，避免冗余
3. 包含使用场景和适用人群
4. 格式：开头（吸引眼球）→ 特点（3 个卖点）→ 结尾（购买理由）

商品名称：{product_name}
关键词：{keywords}
```

**缓存策略**：
- 相似商品描述复用
- 缓存命中率 > 70%
- TTL 72 小时

**成本控制**：
- Token 限制 500/商品
- 单次成本 ¥0.03

### 实施结果

**效率提升**：
- 从 2-3 小时/商品 → 1 分钟/商品（提升 120-180 倍）

**成本降低**：
- 月成本从 ¥500,000 → ¥10,000（降低 98%）
  - API 费用：¥3,000（100,000 商品 × ¥0.03）
  - 服务器成本：¥2,000
  - 维护成本：¥5,000（1 个运维）

**质量**：
- 人工审核通过率 85%（15% 需要修改）

**ROI 计算**：
```
成本：¥10,000/月
收益：¥490,000/月（节省人力成本）
ROI = (490,000 - 10,000) / 10,000 × 100% = 4,800%
```

### 经验总结

1. **缓存是降低成本的关键**（命中率 70% → 成本降低 70%）
2. **Prompt 设计影响质量**（需要迭代优化 5-10 次）
3. **人工审核必不可少**（保留质量把关）
4. **持续监控效果**（定期评估生成质量）
