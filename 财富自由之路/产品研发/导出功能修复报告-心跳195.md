## 根本原因

### 问题定位

通过深入调试 `export.ts` 中的导出逻辑，发现：

```typescript
// ❌ 错误的实现
await new Promise<void>((resolve, reject) => {
  link.addEventListener('click', () => {
    setTimeout(() => {
      document.body.removeChild(link)
      console.log(`图表导出成功: ${filename}`)
      resolve()  // ← 永远不会执行
    }, 100)
  })
})

link.click()
```

**问题分析**：
1. `addEventListener('click', ...)` 事件监听器永远不会被触发
2. Promise 永远不会 resolve，导致 `await` 无限等待
3. 后续图表的导出被阻塞

### 为什么 `addEventListener('click')` 不会触发？

这种通过 JavaScript 创建 `<a>` 标签并触发 `click()` 的方式，在某些浏览器中：
- `click()` 事件可能会被浏览器优化掉
- 事件监听器可能没有被正确注册
- 下载是同步的，但事件监听是异步的

---

## 修复方案

### 正确的实现

```typescript
// ✅ 正确的实现
// 创建下载链接
const link = document.createElement('a')
link.href = dataURL
link.download = `${filename}.${format}`

// 添加到 DOM
document.body.appendChild(link)

// 触发下载（不等待下载完成，直接触发即可）
link.click()

// 等待一小段时间确保下载开始
await new Promise(resolve => setTimeout(resolve, 100))

// 清理 DOM
try {
  document.body.removeChild(link)
} catch (e) {
  // 元素可能已经被移除，忽略
}

console.log(`图表导出成功: ${filename}`)
```

**修复要点**：
1. 移除 `addEventListener('click', ...)` 的逻辑
2. 直接调用 `link.click()` 触发下载
3. 等待 100ms 确保下载开始
4. 清理 DOM 元素
5. **不再等待下载完成**，因为浏览器下载是异步且无法可靠监听的

## 经验总结

### 问题定位的关键

1. **从现象深入到根因**：
   - 初步认为是增长曲线图表的问题
   - 通过日志分析发现是导出逻辑的问题

2. **理解异步编程的陷阱**：
   - `addEventListener` 不会在 `click()` 调用时触发
   - 需要理解浏览器下载的异步特性

3. **简化问题**：
   - 复杂的事件监听逻辑 -> 简单的延迟等待
   - 等待下载完成 -> 等待下载开始

### 技术要点

1. **浏览器下载的异步特性**：
   - 下载是浏览器原生行为，无法通过 JavaScript 可靠监听
   - 不要试图等待下载完成，而是确保下载开始

2. **Promise 的正确使用**：
   - Promise 必须有 resolve 或 reject，否则会永远等待
   - 不要依赖不确定的事件来触发 Promise resolve

3. **调试技巧**：
   - 使用详细的日志追踪执行流程
   - 通过日志定位代码卡住的位置
   - 逐步简化问题，找到最小可复现用例
