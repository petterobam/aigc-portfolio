# 用户反馈机制 v1.0

> 财富自由之路 - MVP 版本
> 创建日期：2026-03-27
> 目标：建立系统化的用户反馈收集机制，为产品迭代提供依据

---

## 一、为什么需要反馈机制

### 1.1 当前问题

**产品已完成，但未激活**：
- MVP 开发完成 100%
- 用户数据录入 0%
- 产品价值未被验证

**缺乏反馈渠道**：
- 没有系统化的反馈收集方式
- 无法了解用户真实需求
- 无法验证产品市场匹配度（PMF）

**优化方向不明确**：
- v1.0 应该优先开发什么功能？
- 用户最痛的需求是什么？
- 哪些功能是多余的？

### 1.2 反馈机制的价值

**对用户**：
- 参与产品成长，获得归属感
- 反馈被重视，提升满意度
- 影响产品方向，获得更好体验

**对产品**：
- 验证假设，避免盲目开发
- 识别核心价值，聚焦关键功能
- 建立用户信任，形成口碑

**对商业**：
- 降低获客成本（口碑传播）
- 提高转化率（满足真实需求）
- 加速 PMF 验证（尽快找到市场）

## 二、反馈渠道设计

### 2.1 应用内反馈（主渠道）

**目标**：让用户在使用过程中最方便地提供反馈

**实现方式**：

#### 2.1.1 反馈入口

**位置 1：看板页面**
- 右上角"反馈"按钮
- 点击后打开反馈弹窗

**位置 2：设置页面**
- "帮助与反馈"菜单项
- 点击后跳转到反馈页面

**位置 3：全局快捷键**
- macOS：Cmd + Shift + F
- Windows：Ctrl + Shift + F

#### 2.1.2 反馈类型

**类型 1：Bug 反馈**
- 问题描述
- 问题截图（自动上传）
- 问题复现步骤
- 系统信息（自动收集）

**类型 2：功能建议**
- 建议描述
- 使用场景
- 期望效果

**类型 3：使用感受**
- 满意度评分（1-5 星）
- 最喜欢的功能
- 最不满意的地方
- 改进建议

**类型 4：数据异常**
- 异常数据描述
- 正确数据是什么
- 系统信息（自动收集）

#### 2.1.3 反馈表单设计

**必填项**：
- 反馈类型（下拉选择）
- 问题描述（文本框，最多 500 字）

**可选项**：
- 联系邮箱（用于跟进）
- 问题截图（最多 3 张）
- 复现步骤（仅 Bug 反馈）

**自动收集**：
- 应用版本号
- 操作系统版本
- 设备信息
- 当前页面
- 使用时间

#### 2.1.4 反馈提交

**存储方式**：
- 本地 SQLite 数据库（表名：feedbacks）
- 自动同步到云端（用户授权后）

**提交后反馈**：
- "感谢您的反馈！我们会尽快处理。"
- 显示反馈编号（如：#FB-20260327-001）
- 预计处理时间（3 个工作日内）

### 2.2 用户测试任务（辅助渠道）

**目标**：引导用户完成特定任务，观察使用行为

#### 2.2.1 测试任务列表

**任务 1：首次使用流程**
- [ ] 打开应用
- [ ] 完成设置向导（3 步）
- [ ] 查看财务看板
- [ ] 点击"记账"按钮，记录一笔支出

**任务 2：修改财务数据**
- [ ] 打开应用，进入设置页面
- [ ] 修改月收入：30,000 → 35,000
- [ ] 修改月支出：10,000 → 8,000
- [ ] 查看看板页面，观察变化
- [ ] 进入目标页面，观察目标重新计算

**任务 3：记账体验**
- [ ] 打开应用，点击"记账"按钮
- [ ] 记录一笔收入（2,000 元）
- [ ] 记录一笔支出（50 元，餐饮）
- [ ] 查看月度报告页面
- [ ] 观察支出结构饼图

**任务 4：目标调整**
- [ ] 进入目标页面
- [ ] 修改梦想月成本：20,000 → 25,000
- [ ] 观察财务自由目标变化
- [ ] 调整年化收益：8% → 10%
- [ ] 观察达成时间变化

**任务 5：理论洞察**
- [ ] 查看每日财富洞察
- [ ] 进入目标页面，查看三个理论洞察
- [ ] 点击"财务保障洞察"查看详细解释
- [ ] 记录最深刻的洞察

#### 2.2.2 任务反馈表

**每个任务完成后填写**：

**问题 1：任务难度**
- 非常简单 ⭐
- 比较简单 ⭐⭐
- 一般 ⭐⭐⭐
- 比较困难 ⭐⭐⭐⭐
- 非常困难 ⭐⭐⭐⭐⭐

**问题 2：任务时间**
- 实际耗时：___ 分钟
- 期望耗时：___ 分钟

**问题 3：遇到的问题**
- 哪个步骤不清楚？
- 哪个操作不顺手？
- 哪个设计不符合预期？

**问题 4：改进建议**
- 如何简化这个任务？
- 如何提升体验？

### 2.3 用户访谈（深度反馈）

**目标**：了解用户深层需求和使用场景

#### 2.3.1 访谈对象

**目标用户**：
- 已经完成设置的用户
- 至少使用 3 天的用户
- 愿意分享真实想法的用户

**用户筛选**：
- 发送访谈邀请邮件
- 提供访谈报酬（50-100 元 / 小时）
- 承诺匿名和保密

#### 2.3.2 访谈问题设计

**第一部分：使用背景**
1. 你目前的财务状况如何？（简单描述）
2. 你对财务自由的了解程度？
3. 你之前用过哪些财务软件？
4. 为什么选择我们的产品？

**第二部分：使用体验**
1. 你最喜欢哪个功能？为什么？
2. 你最不满意哪个功能？为什么？
3. 你觉得哪个功能最有用？
4. 你觉得哪个功能可以删掉？

**第三部分：真实需求**
1. 你最希望产品帮你解决什么问题？
2. 你觉得还缺少什么功能？
3. 你期望产品 1 年后是什么样子？
4. 你愿意为这个产品付费吗？为什么？

**第四部分：改进建议**
1. 你会如何向朋友介绍这个产品？
2. 你会推荐给谁使用？为什么？
3. 你希望产品在哪里改进？
4. 你有什么其他想法？

#### 2.3.3 访谈形式

**方式 1：视频访谈**（推荐）
- 时长：60 分钟
- 工具：腾讯会议 / 钉钉 / Zoom
- 记录：屏幕录制 + 录音

**方式 2：电话访谈**
- 时长：30 分钟
- 工具：微信语音 / 电话
- 记录：录音 + 笔记

**方式 3：问卷访谈**
- 时长：15 分钟
- 工具：问卷星 / 飞书表单
- 记录：在线填写

### 2.4 数据埋点（自动化反馈）

**目标**：通过用户行为数据发现问题

#### 2.4.1 关键事件埋点

**事件 1：应用启动**
- 事件名：app_start
- 参数：时间戳、版本号、操作系统

**事件 2：设置完成**
- 事件名：setup_completed
- 参数：时间戳、设置的月收入、月支出、净资产

**事件 3：记账操作**
- 事件名：transaction_created
- 参数：时间戳、类型（收入/支出）、金额、分类

**事件 4：查看目标**
- 事件名：goals_viewed
- 参数：时间戳、查看的目标（保障/安全/自由）

**事件 5：反馈提交**
- 事件名：feedback_submitted
- 参数：时间戳、反馈类型、反馈内容

#### 2.4.2 关键指标监控

**留存指标**：
- 次日留存率
- 7 日留存率
- 30 日留存率

**活跃指标**：
- 日活跃用户数（DAU）
- 周活跃用户数（WAU）
- 月活跃用户数（MAU）

**功能使用率**：
- 记账功能使用率
- 目标查看率
- 反馈提交率

**异常指标**：
- 启动失败率
- 崩溃率
- 错误率

#### 2.4.3 数据分析

**每日分析**：
- 前一日新增用户数
- 前一日活跃用户数
- 前一日提交反馈数
- 前一日错误数

**每周分析**：
- 本周新增用户数 vs 上周
- 本周留存率 vs 上周
- 本周反馈类型分布
- 本周功能使用率变化

**每月分析**：
- 本月用户增长趋势
- 本月用户留存趋势
- 本月反馈主题分析
- 本月功能使用情况

## 三、反馈处理流程

### 3.1 反馈收集

**实时收集**：
- 应用内反馈提交后，立即存储到数据库
- 自动发送通知给产品负责人

**定期收集**：
- 每周导出所有反馈数据
- 生成反馈统计报告

### 3.2 反馈分类

**分类维度 1：反馈类型**
- Bug 反馈（技术问题）
- 功能建议（新功能需求）
- 体验优化（现有功能改进）
- 使用感受（满意度反馈）
- 数据异常（数据问题）

**分类维度 2：优先级**
- P0 - 严重 Bug（崩溃、数据丢失）
- P1 - 重要 Bug（功能不可用）
- P2 - 一般 Bug（体验问题）
- P3 - 功能建议（新功能需求）
- P4 - 体验优化（细节改进）

**分类维度 3：影响范围**
- 全体用户
- 部分用户
- 单个用户

### 3.3 反馈处理

**P0 - 严重 Bug**
- 响应时间：2 小时内
- 修复时间：24 小时内
- 发布时间：48 小时内
- 通知用户：邮件通知 + 应用内通知

**P1 - 重要 Bug**
- 响应时间：4 小时内
- 修复时间：3 天内
- 发布时间：1 周内
- 通知用户：邮件通知

**P2 - 一般 Bug**
- 响应时间：24 小时内
- 修复时间：1 周内
- 发布时间：2 周内
- 通知用户：版本更新说明

**P3 - 功能建议**
- 响应时间：3 天内
- 评估时间：1 周内
- 排入计划：根据优先级和资源
- 通知用户：邮件通知 + 版本更新说明

**P4 - 体验优化**
- 响应时间：1 周内
- 评估时间：2 周内
- 排入计划：根据优先级和资源
- 通知用户：版本更新说明

### 3.4 反馈闭环

**通知用户**：
- Bug 修复后，通知提交反馈的用户
- 功能上线后，通知提出建议的用户
- 版本更新后，发送更新说明邮件

**感谢用户**：
- 对贡献大的用户，给予奖励
- 在版本更新说明中，致谢贡献者
- 建立用户贡献排行榜

## 四、反馈数据管理

### 4.1 数据库设计

**表名：feedbacks**

| 字段名 | 类型 | 说明 |
|-------|------|------|
| id | INTEGER | 主键，自增 |
| feedback_id | VARCHAR(50) | 反馈编号（如：FB-20260327-001） |
| user_id | VARCHAR(100) | 用户 ID（可选） |
| feedback_type | VARCHAR(20) | 反馈类型（bug/suggestion/feeling/data） |
| priority | VARCHAR(10) | 优先级（P0/P1/P2/P3/P4） |
| title | VARCHAR(200) | 反馈标题 |
| description | TEXT | 反馈描述 |
| attachments | TEXT | 附件（JSON 格式） |
| contact_email | VARCHAR(100) | 联系邮箱（可选） |
| status | VARCHAR(20) | 状态（pending/in_progress/resolved/closed） |
| app_version | VARCHAR(20) | 应用版本号 |
| os_version | VARCHAR(50) | 操作系统版本 |
| device_info | TEXT | 设备信息（JSON 格式） |
| current_page | VARCHAR(100) | 当前页面 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |
| resolved_at | DATETIME | 解决时间 |
| resolver | VARCHAR(50) | 解决人 |

**索引**：
- PRIMARY KEY (id)
- INDEX feedback_id (feedback_id)
- INDEX feedback_type (feedback_type)
- INDEX priority (priority)
- INDEX status (status)
- INDEX created_at (created_at)

### 4.2 数据统计

**按类型统计**：
```sql
SELECT
  feedback_type,
  COUNT(*) as count,
  ROUND(COUNT(*) * 100.0 / (SELECT COUNT(*) FROM feedbacks), 2) as percentage
FROM feedbacks
GROUP BY feedback_type
ORDER BY count DESC;
```

**按优先级统计**：
```sql
SELECT
  priority,
  COUNT(*) as count,
  ROUND(COUNT(*) * 100.0 / (SELECT COUNT(*) FROM feedbacks), 2) as percentage
FROM feedbacks
GROUP BY priority
ORDER BY priority;
```

**按状态统计**：
```sql
SELECT
  status,
  COUNT(*) as count,
  ROUND(COUNT(*) * 100.0 / (SELECT COUNT(*) FROM feedbacks), 2) as percentage
FROM feedbacks
GROUP BY status
ORDER BY status;
```

**按日期统计**：
```sql
SELECT
  DATE(created_at) as date,
  COUNT(*) as count
FROM feedbacks
WHERE created_at >= DATE('now', '-30 days')
GROUP BY DATE(created_at)
ORDER BY date DESC;
```

### 4.3 数据导出

**导出为 CSV**：
```sql

.mode csv
.output feedbacks_20260327.csv

.output
```

**导出为 JSON**：

## 七、风险与应对

### 风险 1：用户不愿意提供反馈

**应对**：
- 简化反馈流程，3 步完成
- 提供奖励机制（如：积分、徽章）
- 强调反馈价值："您的反馈会让产品更好"

### 风险 2：反馈质量不高

**应对**：
- 设计清晰的反馈表单
- 提供反馈示例
- 引导用户描述具体场景

### 风险 3：反馈处理不及时

**应对**：
- 建立反馈处理 SLA
- 设置自动提醒机制
- 优先处理 P0/P1 反馈

### 风险 4：反馈数量过多

**应对**：
- 自动分类和优先级排序
- 使用 AI 辅助分析反馈
- 建立反馈处理团队

---

## 八、核心原则

1. **用户价值优先**
   - 反馈机制要方便用户，而不是方便开发者
   - 反馈处理要及时，让用户感受到被重视

2. **数据驱动**
   - 基于反馈数据做决策，而不是凭感觉
   - 定期分析反馈趋势，发现问题和机会

3. **快速响应**
   - 及时回复反馈，建立信任
   - 快速修复 Bug，提升满意度

4. **持续优化**
   - 反馈机制本身也需要优化
   - 根据用户反馈改进反馈机制

5. **建立信任**
   - 告诉用户反馈会被重视
   - 透明化反馈处理流程
   - 感谢用户的反馈
